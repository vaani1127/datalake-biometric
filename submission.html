1. Problem Statement

"How can we accurately and securely authenticate field personnel using facial recognition and liveness detection on standard mid-range mobile devices without any active internet connection, while ensuring the AI model remains lightweight and seamlessly integrates with a React Native application on both Android and iOS devices?"

NHAI field staff operate in remote highway construction sites where network connectivity is unreliable or entirely unavailable. Traditional biometric systems require cloud connectivity, making them unsuitable for these zero-network zones. Manual attendance is error-prone and susceptible to proxy attendance fraud.

2. Our Solution — Datalake Biometric SDK

A React Native TurboModule that runs entirely on-device — no internet required during enrolment or verification. All AI inference, encrypted storage, liveness detection and attendance logging happen locally. When connectivity is restored, signed records sync to AWS and local data is purged.

3. AI Model Architecture

3.1 Model Pipeline — Verification Flow

1
Image Capture (Vision Camera) — Selfie JPEG via takePhoto(), EXIF rotation applied. Downscaled to 720px long-side to prevent OOM on 12 MP cameras.
2
Quality Gate — Laplacian variance (blur score) + brightness luma range check (40–220). Combined score < 0.5 → returns POOR_QUALITY instantly, no model inference.
3
Face Detection — BlazeFace (0.22 MB) — MediaPipe short-range TFLite model. ML Kit face box hint fed from JS frame processor for tight crop. Falls back to centre-biased heuristic if no hint.
4
Crop & Resize — Detected face region cropped with 20% padding, scaled to 112×112 pixels (MobileFaceNet input size).
5
Face Embedding — MobileFaceNet INT8 (5.0 MB) — Produces a 512-dimensional L2-normalised embedding vector. XNNPACK delegate, 4 threads. ~28–31 ms on Exynos 1330.
6
1-N Cosine Matching — Query embedding dotted against all enrolled vectors in SQLCipher DB. Cosine similarity > 0.65 → MATCH. Returns worker ID + confidence score.

3.3 INT8 Quantization — How We Stay Under 20 MB

MobileFaceNet float32 baseline is ~18 MB. Post-training INT8 quantization maps each weight to an 8-bit integer with a per-channel scale and zero-point. At inference, values are dequantized back to float for the dot-product:

// TFLiteEngine.kt — INT8 dequantization
val q = outputTensor.quantizationParams()
val embedding = FloatArray(outDim) { i ->
    val raw = (out[0][i].toInt() and 0xFF)
    (raw - q.zeroPoint) * q.scale   // real_value = (quant - zeroPoint) * scale
}

Result: 5.0 MB on disk, ~8 MB resident in memory — comfortably within mid-range device constraints. Accuracy loss vs. float32: <0.5% TAR.

4. Liveness Detection — Multi-Challenge Anti-Spoofing

A static photograph or looped video cannot complete any of the three challenges below because they require real-time motor response from a live person. The challenge is randomly selected each session to prevent replay attacks.

👁️
Blink Challenge
2 open → closed → open eye transitions required within 12 seconds.

Signal: ML Kit leftEyeOpenProbability + rightEyeOpenProbability
Threshold: closed < 0.3, open > 0.7
😊
Smile Challenge
A genuine smile must be held until the sticky flag is set.

Signal: ML Kit smilingProbability
Threshold: > 0.7 (70% probability of smile)
Sticky: once detected, stays true until reset.
↔️
Head-Turn Challenge
Head must be turned left or right beyond ±20°.

Signal: ML Kit headEulerAngleY (yaw)
Threshold: |yaw| > 20 degrees
Sticky: once detected, stays true until reset.

4.1 Frame Processor — Real-Time Detection (camera.ts)

// Runs in a native worklet — zero JS bridge overhead per frame
const frameProcessor = useFrameProcessor((frame) => {
  'worklet';
  const faces = detectFaces(frame);           // ML Kit on-device, offline
  const f = faces[0];
  const left   = f.leftEyeOpenProbability;    // 0.0 (closed) → 1.0 (open)
  const right  = f.rightEyeOpenProbability;
  const smiling = f.smilingProbability;        // 0.0 → 1.0
  const yaw    = f.headEulerAngleY;            // degrees, – = left, + = right
  onFace(faces.length, left, right, nx, ny, nw, nh, smiling, yaw);
}, [detectFaces, onFace]);

4.3 Native EAR Fallback (Apache 2.0 — fully open source)

The native LivenessEngine.kt implements Eye Aspect Ratio (EAR) detection using MediaPipe 468-point Face Mesh landmarks — a fully open-source path that can replace the ML Kit path when needed:

// LivenessEngine.kt — EAR formula (MediaPipe landmark indices)
EAR = (dist(p1,p5) + dist(p2,p4)) / (2 * dist(p0,p3))
// EAR < 0.20 → eyes closed. Requires ≥2 blinks AND EAR variance > 0.0005
// (Low variance = looped video; real eyes have micro-variation every frame)

5. Offline Capability & Security Architecture

Zero network calls during enrolment or verification. All TFLite/CoreML inference, SQLCipher reads/writes, and HMAC signing happen entirely on-device. The app operates correctly with Airplane Mode permanently on.
🔐 SQLCipher AES-256-CBC

Entire biometric.db file is encrypted at rest. Passphrase (32 bytes, random) is generated on first launch, stored only in Android EncryptedSharedPreferences (encrypted by Keystore AES-256-GCM master key). An attacker with raw filesystem access cannot read embeddings.

🗝️ Android Keystore / iOS Keychain

DB passphrase and HMAC key are bound to the app's Keystore entry. Keys are destroyed on app uninstall. Never appear in plaintext in memory beyond the duration of a single DB open call. iOS equivalent uses CryptoKit + Keychain with kSecAttrAccessibleAfterFirstUnlockThisDeviceOnly.

✍️ HMAC-SHA256 Attendance Signing

Every attendance record is signed before writing to DB: workerId | timestamp | lat | lng | confidence | deviceId → HMAC-SHA256(Keystore-backed key). AWS Lambda verifies signature before accepting into DynamoDB, preventing record tampering in transit.

🚫 No Raw Biometrics Stored

Camera frames are never written to disk. Only the 512-dimensional L2-normalised embedding (mathematical representation) is stored — it is irreversible: you cannot reconstruct a face image from an embedding. Compliant with DPDP Act 2023 data minimisation principles.

6. Sync & Purge Mechanism

6.1 Offline → Online Flow

1
Queue (offline) — Each verified attendance is HMAC-signed and inserted into attendance_log with synced=0. No network required.
2
Auto-detect connectivity — @react-native-community/netinfo fires on offline→online transition. Pending records are fetched and sync starts automatically.
3
Batch POST to AWS Lambda — All pending records sent as a JSON array to the sync endpoint. Each record carries its HMAC-SHA256 signature.
4
Lambda validates & writes DynamoDB — Idempotent write (attribute_not_exists(id)). Returns per-record status: stored or duplicate. Duplicates are safe to retry.
5
Mark synced + Purge local DB — On server ACK, markSynced(ids) sets synced=1, then purgeSyncedRecords() runs DELETE FROM attendance_log WHERE synced=1. Local storage is fully cleaned.

6.2 AWS Backend Architecture

ComponentTechnologyPurpose
Sync endpoint AWS Lambda (Python 3.11) + API Gateway Receive, validate, and store attendance records
Database Amazon DynamoDB Persistent attendance store, GSI for worker/date queries
Auto-expiry DynamoDB TTL (90 days) Cloud-side purge — no manual cleanup needed
IaC AWS SAM (template.yaml) One-command deploy: sam deploy

6.3 Idempotency — Safe Retries on Connectivity Drops

# backend/lambda_sync_handler.py
table.put_item(
    Item={ "id": record["id"], "workerId": record["workerId"], ... },
    ConditionExpression="attribute_not_exists(id)"  # reject duplicates silently
)
# Returns: { "status": "stored" }  or  { "status": "duplicate" }
# JS marks record as synced for BOTH statuses — no double-write, no data loss

8. Integration into Datalake 3.0

Step 1 — Install the SDK

yarn add datalake-biometric
# iOS: cd ios && bundle exec pod install

Step 2 — Initialize on App Start

import { BiometricSDK } from 'datalake-biometric';

// Call once on app launch — loads TFLite models and opens SQLCipher DB
await BiometricSDK.initialize();

Step 3 — Enroll a Worker (3 frames)

// Capture 3 frames from Vision Camera, convert to base64
const frames = [base64Frame1, base64Frame2, base64Frame3];

const result = await BiometricSDK.enrollWorker('EMP-1042', frames);
// → { success: true, framesUsed: 3 }
// Embedding averaged + L2-normalized, stored in encrypted SQLCipher DB

Step 4 — Verify with Liveness (Datalake integration point)

// 1. Show camera with frame processor running multi-challenge liveness
const { frameProcessor, blinkCount, smileDetected, headTurned } = useFaceState();

// 2. Once liveness challenge passes → capture photo and verify
const b64 = await takePhotoBase64(cameraRef.current);
const result = await BiometricSDK.verifyWorker(b64);

// result.status: 'MATCH' | 'NO_MATCH' | 'NO_FACE' | 'POOR_QUALITY'
// result.workerId: 'EMP-1042'     (if MATCH)
// result.confidence: 0.92         (cosine similarity, 0–1)
// result.inferenceMs: 31          (TFLite inference only)
// result.totalMs: 487             (full pipeline)

Step 5 — Log Attendance & Queue for Sync

if (result.status === 'MATCH') {
    await BiometricSDK.logAttendance(
        result.workerId,
        currentLatitude,    // GPS coords from device
        currentLongitude,
        result.confidence
    );
    // Record is HMAC-signed and queued locally — no network needed
}

Step 6 — Sync to AWS (when connectivity restored)

// Auto-fires on offline→online transition via NetInfo listener
const pending = await BiometricSDK.getPendingRecords();
const response = await fetch(SYNC_ENDPOINT, {
    method: 'POST',
    body: JSON.stringify({ records: pending })
});
const { results } = await response.json();
const syncedIds = results.filter(r => r.status === 'stored' || r.status === 'duplicate')
                         .map(r => r.id);
await BiometricSDK.markSynced(syncedIds);
await BiometricSDK.purgeSyncedRecords();  // Wipes local DB — data purged ✓

Step 7 — Deploy AWS Backend (one command)

cd backend
sam build && sam deploy --guided
# Creates: API Gateway + Lambda + DynamoDB table with 90-day TTL
